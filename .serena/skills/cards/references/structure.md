# Структура ответа на карточку

## Идеальная структура (4 блока)

```
┌─────────────────────────────────────────────────────────────┐
│ 1. ПРОСТОЕ ОПРЕДЕЛЕНИЕ                                      │
│    1-2 предложения своими словами                           │
│    "Это способ делать X..."                                 │
├─────────────────────────────────────────────────────────────┤
│ 2. МЕХАНИЗМ / ПРИНЦИП РАБОТЫ                                │
│    Как это работает под капотом                             │
│    "Проще говоря, когда происходит Y, то..."               │
├─────────────────────────────────────────────────────────────┤
│ 3. ПРИМЕР ИЗ ПРОЕКТА                                        │
│    "В Task Manager я использовал это для..."               │
│    Конкретная ситуация, не абстракция                       │
├─────────────────────────────────────────────────────────────┤
│ 4. ПРАКТИЧЕСКАЯ ПОЛЬЗА                                      │
│    "Это позволяет..." / "Благодаря этому..."               │
│    Почему это важно на практике                             │
└─────────────────────────────────────────────────────────────┘
```

## Примеры правильных ответов

### Пример 1: Goroutines

**Блок 1 (Определение):**
> Горутины — это способ в Go запускать задачи параллельно. Это не потоки ОС, а легковесные сопрограммы.

**Блок 2 (Механизм):**
> Проще говоря, когда запускаешь функцию с `go`, она начинает выполняться независимо от основного потока. Go runtime сам распределяет горутины по потокам ОС.

**Блок 3 (Пример из проекта):**
> В Task Manager я использовал горутины для параллельной обработки нескольких задач одновременно — вместо того чтобы ждать последовательно, запускал каждую в своей горутине и собирал результаты через каналы.

**Блок 4 (Польза):**
> Это позволяет обрабатывать сотни запросов одновременно, не тратя ресурсы на тяжёлые потоки ОС. Горутина занимает всего 2KB памяти против 1MB у потока.

### Пример 2: Context

**Блок 1 (Определение):**
> Context — это способ передавать дедлайны, cancellation сигналы и метаданные между функциями и горутинами.

**Блок 2 (Механизм):**
> То есть когда создаёшь context.WithTimeout, ты говоришь: "у тебя есть 5 секунд, потом отменяйся". И все операции с этим контекстом узнают об отмене и могут корректно завершиться.

**Блок 3 (Пример из проекта):**
> В RAG-сервисе я использовал context для таймаута запросов к LLM. Если модель не отвечала за 30 секунд — контекст отменялся, и клиент получал ошибку, а не ждал вечно.

**Блок 4 (Польза):**
> Благодаря этому система остаётся отзывчивой даже при проблемах с внешними сервисами. Можно гарантировать SLA и не тратить ресурсы на бесконечное ожидание.

## Антипримеры (чего избегать)

### ❌ Слишком абстрактно
> "Context предоставляет механизм для управления жизненным циклом операций..."

### ❌ Книжный стиль
> "Обеспечивает легковесную конкурентность для одновременного выполнения множества функций..."

### ❌ Без примера
> "Goroutines позволяют делать параллельные вычисления. Они эффективны и масштабируемы."

### ❌ С кодом в тексте
> "Используй `go func() { db.Query(...) }()` для запуска горутины..."

## Чек-лист проверки ответа

- [ ] Можно произнести на одном дыхании без запинки?
- [ ] Нет "книжных" слов (механизм, обеспечение, реализация)?
- [ ] Есть связки "Проще говоря", "То есть"?
- [ ] Пример из Task Manager или RAG, не абстрактный?
- [ ] Объяснено, почему это полезно?
- [ ] Нет кода в тексте?
